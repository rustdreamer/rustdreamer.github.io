部署的演变之路

Apr 24, 2018

请注意，这个文章是写给了解部署环境的读者，并没有相关干货提供，谢谢大家支持。

第一阶段：传统安装

下面以 java 为例子（因为第一个读者公司的服务器可能是 java，但我不怎么使用 java 难免有错，如果错误，请联系本人修正，以免误导他人）

  1. 远程连接服务器
  2. 上传数据库、jdk、tomcat各类安装包（受限带宽因素，往往很耗时），上传方式有 ftp/sftp, ssh 等等
  3. 安装数据库
  4. 安装jdk、tomcat、nginx（linux环境下，使用写好的sh脚本直接运行，依然比较耗时）人工部署war包等等

此方式缺点太多，往往需要3-5人天才能完成所有部署，而且服务器较多的时候，需要一台一台的安装环境，所以聪明（懒惰）的程序员想出了更好的方案

第二阶段：系统镜像恢复

  1. 部署好单台服务器，然后制作成镜像
  2. 上传镜像，通过类似阿里云的服务厂商使用系统镜像部署

此方法依旧有很多弊端，服务器配置不同，环境不同，服务器启动故障维护等等还是依赖人工等等，成本较大。是的，更聪明的程序员想到了更好的方案

第三阶段：docker 集群部署

  1. 制作好 docker 镜像（即把业务代码等等都打包成 📦 docker 镜像）
  2. 上传镜像
  3. 集群监听镜像更新后自动拉取镜像，并以最新的镜像启动服务器

这种方式抹平了环境差异带来的问题，并且不需要服务器安装各种软件（软件版本之间还可能引起问题），部署也是非常快速，但是还是需要人工来做一些维护工作，比如伸缩服务器等等，终于人的懒惰爆发了

第四阶段：无服务器部署

无服务器不是没有服务器，而是不需要程序员人工部署、伸缩服务器。而且大多厂商使用按量收费，非常满足公司的需求（毕竟很多公司大晚上的服务器都是闲置，而很多公司的服务器晚上才开始活跃，这样节省的资源可不是一点点）。
无服务器是一个未来的趋势，很多公司都已经投入实践。
